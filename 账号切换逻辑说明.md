# Codex 账号切换逻辑说明

这份文档解释这个项目为什么能够“管理 Codex 账号并进行切换”，以及它实际做了什么。

## 一句话结论

这个项目并不是调用 OpenAI 的某个“切换账号 API”来完成账号切换。

它的核心做法很直接：

1. 读取当前正在使用的 `~/.codex/auth.json`
2. 把不同账号的认证配置分别保存成多个 JSON 文件
3. 当用户选择切换时，用目标账号的认证内容覆盖 `~/.codex/auth.json`
4. Codex CLI 或依赖这个认证文件的本地程序，之后就会以新的认证信息运行

也就是说，本项目本质上是一个“本地认证文件管理器”。

## 关键前提

这个项目成立的前提是：

- Codex 当前登录态保存在本地文件 `~/.codex/auth.json`
- 这个文件里包含了可复用的认证信息，例如：
  - `auth_mode`
  - `OPENAI_API_KEY`
  - `tokens`
  - `last_refresh`
- Codex 在运行时会读取这个文件，因此只要把这个文件替换成另一个账号的内容，就等于把本地活跃账号切掉了

项目里真正被切换的“核心对象”不是 UI 状态，也不是数据库记录，而是这个本地认证文件。

## 关键路径

### 1. 系统正在使用的认证文件

项目把系统正在生效的 Codex 认证文件固定为：

- `~/.codex/auth.json`

对应实现：

- `config_utils.py` 中 `get_config_paths()` 返回 `system_auth_file = Path.home() / ".codex" / "auth.json"`
- `codex-tauri-app/src/main.js` 中 `initPaths()` 返回 `systemAuthFile = ~/.codex/auth.json`

### 2. 项目自己管理的账号仓库

项目不会只依赖一个当前认证文件，而是会把多个账号副本保存在自己的管理目录里。

Python/Tauri 两边都统一使用：

- macOS: `~/Library/Application Support/com.codex.switch/codex-config/`
- Windows: `%APPDATA%/com.codex.switch/codex-config/`
- Linux: `$XDG_CONFIG_HOME/com.codex.switch/codex-config/` 或 `~/.config/com.codex.switch/codex-config/`

这个目录下面主要有：

- `accounts/`：每个账号一份 JSON 备份
- `usage_cache/`：按邮箱保存的用量缓存

对应实现：

- `config_utils.py`
- `codex-tauri-app/src/main.js`

## 账号是怎么“保存起来”的

### Python 版本

备份逻辑在：

- `backup_current_account.py`
- `codex_account_manager.py`

核心步骤是：

1. 从 `~/.codex/auth.json` 读取当前认证配置
2. 解析 token，尽量从 JWT 里提取邮箱
3. 用邮箱生成账号名
4. 往保存记录里补管理字段：
   - `saved_at`
   - `account_name`
5. 把完整认证快照和管理信息一起存到 `accounts/<账号名>.json`

这里的“账号管理”本质上就是把多个 `auth.json` 版本备份成多个独立文件。

### Tauri 桌面版

桌面版对应逻辑在：

- `codex-tauri-app/src/main.js`

`quickSave()` 做的事情和 Python 版一致：

1. 读取 `~/.codex/auth.json`
2. 解析邮箱
3. 生成账号名
4. 生成一份“完整认证快照 + 管理元数据”的账号记录
5. 保存到应用自己的 `accounts` 目录

## 账号是怎么“切换”的

这是整个项目最关键的部分。

### Python 版切换逻辑

见：

- `switch_account.py`
- `codex_account_manager.py`

核心步骤：

1. 找到目标账号文件：`accounts/<账号名>.json`
2. 读取这份账号配置
3. 从账号记录中提取出原始认证快照
4. 去掉仅属于账号管理器的附加字段
5. 把这份“原始认证配置”写回 `~/.codex/auth.json`

换句话说，切换账号时并不会请求网络，也不会重新登录。
它只是把“当前系统正在使用的认证文件内容”替换成另一个账号保存下来的内容。

### Tauri 版切换逻辑

桌面版在 `codex-tauri-app/src/main.js` 的 `quickSwitchAccount(accountName)` 里做的是同一件事：

1. 在内存里的 `accounts` 数组中找到目标账号
2. 取出其中的 `config`
3. 从保存记录中提取完整的认证快照
4. 校验这份快照仍然包含有效认证字段
5. 直接写入 `PATHS.systemAuthFile`，也就是 `~/.codex/auth.json`
6. 重新加载账号列表，通过 `account_id`、token、邮箱综合比对把“当前账号”高亮出来

所以“切换成功”的定义，不是服务器返回成功，而是本地文件已经被覆盖成功。

## 它怎么知道哪个账号是“当前账号”

项目并没有单独维护一个“当前账号指针”。

现在的判断方法更稳：

1. 先读取当前系统文件 `~/.codex/auth.json`
2. 优先提取 `account_id`
3. 如果没有稳定的账号 ID，再比较 `refresh_token` / `id_token` / `access_token`
4. 最后才退化到邮箱比较
5. 在 `accounts/*.json` 里找到最匹配的那一项，视为“当前账号”

对应实现：

- Python: `usage_checker.py` 的 `extract_email_from_auth()`
- Tauri: `codex-tauri-app/src/main.js` 的 `parseJwtPayload()`、`extractEmailFromToken()`

这说明项目现在优先用账号 ID 和 token 识别当前账号，邮箱只是最后的兜底。

## 为什么 Tauri 桌面版有权限直接读写这些文件

桌面版不是普通网页，它运行在 Tauri 容器里。

Rust 壳层在 `codex-tauri-app/src-tauri/src/lib.rs` 注册了：

- `tauri_plugin_fs`
- `tauri_plugin_dialog`

同时权限文件 `codex-tauri-app/src-tauri/capabilities/default.json` 明确放开了这些目录：

- `$HOME/.codex/**`
- `$APPDATA/codex-config/**`

并允许这些文件系统操作：

- 读文本文件
- 写文本文件
- 列目录
- 创建目录
- 删除文件
- 查看文件状态

因此桌面版能够：

- 直接读取 `~/.codex/auth.json`
- 直接扫描 `~/.codex/sessions`
- 直接把切换后的配置写回系统目录

这就是它“能切号”的桌面权限基础。

## 用量信息是怎么来的

这个项目除了切号，还显示用量。这里的逻辑也不是请求 OpenAI 账户后台接口。

它读取的是本地 Codex 会话日志：

- `~/.codex/sessions/**/rollout-*.jsonl`

流程是：

1. 找出最近的 `rollout-*.jsonl`
2. 优先选包含 `token_count` 事件的文件
3. 从后往前扫描，找到最新的 `payload.type == "token_count"` 记录
4. 提取：
   - `info.total_token_usage`
   - `payload.rate_limits`
5. 把结果按邮箱缓存到 `usage_cache/`

对应实现：

- Python: `usage_checker.py`
- Tauri: `codex-tauri-app/src/main.js`

这也解释了为什么项目里会提示“先用 codex 发一条消息再刷新用量”：

- 因为没有新的 session 记录，就没有新的 `token_count` 事件可读

## 这个项目真正管理的是什么

严格地说，这个项目管理的是三类本地数据：

1. 当前生效认证
   - `~/.codex/auth.json`
2. 多账号备份
   - 应用配置目录下的 `accounts/*.json`
3. 用量缓存
   - 应用配置目录下的 `usage_cache/*.json`

所以它的能力边界非常明确：

- 它能切换本地保存过的账号
- 它能把当前账号另存为一个备份
- 它能根据本地 session 文件推断最近用量

但它不能：

- 代替真实登录流程
- 帮你获取一个新的账号 token
- 让已经失效的 token 重新有效
- 绕过服务端鉴权

## 可以把它理解成什么

可以把它理解成一个专门面向 Codex 的“认证配置版本管理器”：

- `auth.json` 是当前工作副本
- `accounts/*.json` 是历史快照
- “切换账号”就是把某个快照恢复成当前工作副本

这个思路和很多开发工具中的“profile 切换”“凭证文件切换”是一样的，只是这里的目标文件是 Codex 的认证文件。

## 这套实现的局限和风险

### 1. 项目依赖 `auth.json` 的结构稳定

旧实现默认认为少数字段就足够，这会在 Codex 新增关键字段时产生兼容性问题。

现在项目会保存并恢复完整认证快照，因此对 `auth.json` 结构变化的容忍度更高。
不过如果未来 Codex 改成把关键登录态放到别处，而不再依赖 `~/.codex/auth.json`，这套方案仍然需要调整。

### 2. 当前账号识别仍然依赖本地可读标识

虽然现在已经优先比较 `account_id` 和 token，但前提仍然是这些信息能从本地认证数据中读出来。
如果未来这些标识被移除或迁移，项目仍然需要同步适配。

### 3. 它切的是本地状态，不是远端会话

这意味着：

- 本地文件切换成功，不代表服务端一定接受这个 token
- 如果 token 已过期，切过去也还是不能正常用
- 某些已经运行中的进程可能需要重启后才会完全使用新配置

### 4. 用量查询不是按账号实时调用接口

它读取的是本地 session 产物，再结合邮箱缓存展示。
所以非当前账号的用量，很多时候展示的是历史缓存，不一定是实时值。

## 最后总结

这个项目之所以能管理和切换 Codex 账号，核心不是“账号系统集成”，而是下面这条链路：

1. 发现 Codex 当前账号由 `~/.codex/auth.json` 决定
2. 把不同账号的完整认证快照备份到应用自己的 `accounts` 目录
3. 切换时把目标账号的完整认证快照重新写回 `~/.codex/auth.json`
4. 用 `account_id`、token 和邮箱综合识别当前账号
5. 用读取 `~/.codex/sessions` 的方式补充用量展示

因此，这个项目的本质可以概括为：

**通过备份和替换本地认证文件，实现 Codex 多账号管理与快速切换。**
